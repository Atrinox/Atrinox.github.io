## 서론
변수, 함수, 클래스 이름은 코드의 첫인상이다.  
의미 있는 이름은 협업과 유지보수의 핵심이며, 개발자가 반드시 익혀야 할 기본기다.

---

## 의도를 드러내라
```java
int d;              // ❌ 의미 불명확
int elapsedDays;    // ✅ 의도 드러남
````

의도가 드러나는 이름은 코드 이해와 변경을 쉽게 만든다.

```java
public List<Cell> getFlaggedCells() {
    List<Cell> flagged = new ArrayList<>();
    for (Cell cell : gameBoard)
        if (cell.isFlagged())
            flagged.add(cell);
    return flagged;
}
```

---

## 그릇된 정보를 피하라

`hp` 같은 축약은 “health point”로 오해될 수 있다.
`account` vs `accnt`, `controller` vs `control`처럼 비슷한 단어 혼용도 혼란을 준다.
👉 정확하고 오해 없는 단어를 선택하라.

---

## 의미 있게 구분하라

단순한 숫자(`a1`, `a2`)나 불용어(`data`, `info`)는 피하라.

```java
public static void copyChars(char[] src, char[] dest) {
    for (int i = 0; i < src.length; i++) dest[i] = src[i];
}
```

---

## 불용어를 피하라

* `CustomerData` → `Customer`
* `ProductInfo`, `ProductData` → 차이 불분명
  👉 불용어는 의미를 추가하지 않고, 오히려 검색과 유지보수를 어렵게 한다.

---

## 발음·검색하기 쉬운 이름

* `genymdhms` ❌ → `generationTimestamp` ✅
* 문자 하나, 매직 넘버는 검색 불가

```java
const int WORK_DAYS_PER_WEEK = 5;
for (int j = 0; j < NUMBER_OF_TASKS; j++) { ... }
```

---

## 인코딩을 피하라

헝가리안 표기법, 멤버 변수 접두어(`m_`, `_`), `IShape`·`ShapeImpl` 같은 이름은 불필요하다.
👉 `Shape`, `Circle`, `JwtAuthenticationService`처럼 **도메인 용어와 역할**을 드러내라.

---

## 기억력을 시험하지 마라

```java
int r; // report? result?
calculateX(); // 무엇을 계산?
```

👉 `reportCount`, `calculateTax()`처럼 읽자마자 이해되도록 한다.

---

## 클래스·메서드 네이밍

* **클래스 = 명사(구)**: `Order`, `InvoiceRepository`
* **메서드 = 동사(구)**: `calculateTax()`, `sendEmail()`
* 조회/변경을 동사로 명확히 구분한다.

---

## 생성자 오버로드 대신 정적 팩토리

생성자보다 의도 있는 이름을 제공한다.

```java
public static Money of(long amount, Currency cur) { ... }
public static Money zero(Currency cur) { ... }
```
---

## 기발한 이름은 피하라

이름을 창의적으로 짓는 것은 오히려 독자가 의도를 파악하기 어렵게 만든다.  
코드는 농담이 아니라 **정확히 의도를 전달하는 글**이어야 한다.

- ❌ `HolyHandGrenade` (무슨 역할인지 알 수 없음)  
- ✅ `DeleteInactiveUsers` (의도가 명확함)

👉 웃기거나 은유적인 이름 대신, **있는 그대로의 의미**를 표현하라.

---


## 한 개념에 한 단어만 사용하라

같은 개념은 항상 같은 단어로 표현해야 한다.
의미가 같은데 여러 단어를 섞어 쓰면 독자는 **서로 다른 개념인지** 헷갈리게 된다.

### 잘못된 예시

```java
getUserInfo();
fetchUserData();
retrieveUserRecord();
```

→ 모두 “사용자 정보”를 가리키지만, 단어가 달라 혼란을 만든다.

### 개선된 예시

```java
getUser();
fetchUser();
retrieveUser();
```

→ 같은 개념을 한 단어(`User`)로 통일했다.

👉 `manager`, `controller`, `driver` 같은 유사 단어를 뒤섞지 말고, 팀 내에서 **공통 용어 사전**을 정해 일관성을 지켜라.

---

## 말장난을 하지 마라

같은 맥락에서 같은 단어는 같은 의미로만 사용해야 한다.
코드에서 단어를 의도적으로 바꿔 쓰거나, 다른 의미로 활용하는 식의 **말장난은 혼란**을 부른다.

### 잘못된 예시

```java
class Product { ... }
class ProductInfo { ... }   // 사실상 Product와 같은 개념
class ProductData { ... }   // 구분할 수 없는 이름
```

### 개선된 예시

```java
class Product { ... }
class ProductRepository { ... }   // 저장소 역할을 드러냄
class ProductService { ... }      // 서비스 역할을 드러냄
```

👉 같은 개념은 같은 단어로, 다른 개념은 확실히 다른 단어로 표현하라.
이름에서 말장난을 피하고, **직관적이고 일관된 용어**를 선택해야 한다.

---

## 해당 영역에서 가져온 이름을 사용하라

이름은 코드가 속한 **도메인 영역의 언어**를 반영해야 한다.
비즈니스 로직을 표현할 때 일반적인 프로그래밍 용어나 추상적인 단어 대신,
실제 업무/문제 영역에서 사용하는 **도메인 용어**를 그대로 쓰는 것이 가장 명확하다.

### 잘못된 예시

```java
int d;        // 날짜? 기간? 의미 불분명
int v;        // 값? 변수? 용도 불명확
```

### 개선된 예시

```java
int elapsedDays;
int daysSinceCreation;
Order order = orderRepository.findById(orderId);
```

👉 도메인에서 통용되는 단어를 사용하면 **의도와 맥락이 즉시 드러나고**,
새로 합류한 팀원도 별도의 해설 없이 코드를 이해할 수 있다.

---

## 문제 영역에서 가져온 이름을 사용하라

코드의 이름은 구현 세부사항이 아니라 **문제가 속한 영역(Problem Domain)** 에서 가져와야 한다.
해결하려는 문제 자체의 언어를 사용하면 코드가 **요구사항과 직접적으로 연결**되고,
비즈니스 담당자나 도메인 전문가와도 쉽게 소통할 수 있다.

### 잘못된 예시

```java
int d;              // 날짜와 관련된 무엇인지 알 수 없음
String tmp;         // 임시 데이터? 용도를 알 수 없음
```

### 개선된 예시

```java
int daysSinceCreation;
Money orderAmount;
Customer customer = customerRepository.findById(customerId);
```

👉 문제 영역의 용어를 코드에 반영하면, **구현과 도메인을 잇는 다리**가 되어
설계가 명확해지고 유지보수도 쉬워진다.

---

## 의미 있는 맥락을 추가하라

이름 하나만으로는 불충분할 때가 있다.
그럴 땐 **필요한 맥락을 함께 담아** 의미를 명확히 해야 한다.
단, 쓸데없이 장황하게 늘리는 것이 아니라, **혼란을 피할 만큼만** 추가한다.

### 잘못된 예시

```java
int state;      // 어떤 상태인지 알 수 없음
String name;    // 사용자 이름? 상품 이름?
```

### 개선된 예시

```java
int deliveryState;
String customerName;
String productName;
```

👉 이름은 짧을수록 좋지만, **맥락을 잃으면 오해를 낳는다.**
필요하다면 맥락을 포함시켜서, 코드를 읽는 사람이 **즉시 의미를 이해할 수 있게** 만들어라.

---

## 불필요한 맥락을 없애라

맥락은 의미를 명확히 하지만, **불필요하게 중복된 맥락**은 오히려 코드 가독성을 해친다.
모든 이름에 같은 접두어나 접미사를 붙이는 식은 낭비일 뿐이다.

### 잘못된 예시

```java
class ProductInfo { ... }
class ProductInfoRepository { ... }
class ProductInfoService { ... }
```

### 개선된 예시

```java
class Product { ... }
class ProductRepository { ... }
class ProductService { ... }
```

👉 의미 전달에 기여하지 않는 맥락은 과감히 제거하라.
**필요한 맥락은 유지하고, 중복된 맥락은 줄여야** 코드가 간결하고 읽기 쉬워진다.

## 정리

클린 코드에서 말하는 **의미 있는 이름**의 원칙은 다음과 같이 정리된다.

1. **의도를 드러내라** – 이름만 보고도 목적을 알 수 있어야 한다.
2. **그릇된 정보를 피하라** – 오해될 만한 축약어나 비슷한 단어를 혼용하지 말라.
3. **의미 있게 구분하라** – 숫자, 불용어(`data`, `info`) 같은 모호한 구분은 피하라.
4. **불용어를 피하라** – 이름 길이만 늘리고 의미 없는 단어는 제거하라.
5. **발음·검색하기 쉬운 이름을 쓰라** – 팀 대화와 검색에서 혼란이 없어야 한다.
6. **인코딩을 피하라** – 타입, 범위, 구현 여부 같은 메타정보를 이름에 담지 말라.
7. **기억력을 시험하지 마라** – 축약 대신 읽자마자 이해되는 이름을 사용하라.
8. **클래스=명사, 메서드=동사** – 개념은 명사, 행위는 동사로 표현한다.
9. **생성자 오버로드 대신 정적 팩토리 메서드** – 의도가 드러나는 이름으로 다양한 생성 시나리오를 지원한다.
10. **기발한 이름은 피하라** – 농담·은유 대신 있는 그대로의 의미를 표현하라.
11. **한 개념에 한 단어만 사용하라** – 같은 개념은 항상 같은 용어로 통일한다.
12. **말장난을 하지 마라** – 같은 단어는 같은 의미로만 사용한다.
13. **해당 영역에서 가져온 이름을 사용하라** – 도메인 언어를 코드에 반영하라.
14. **문제 영역에서 가져온 이름을 사용하라** – 문제 자체의 용어를 사용해 요구사항과 직접 연결하라.
15. **의미 있는 맥락을 추가하라** – 혼란을 막기 위해 필요한 맥락을 적절히 포함시켜라.
16. **불필요한 맥락을 없애라** – 중복된 접두어·접미어는 제거해 간결함을 유지하라.
